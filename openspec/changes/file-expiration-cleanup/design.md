## Context

FastTask 的文件系统位于 `./files/` 目录下。用户通过 `/upload` 接口上传文件，通过 `/download` 接口下载文件。`files/fasttask/` 子目录是系统内部目录（存放 Redis 数据、Celery 状态、SSL 证书、日志、配置等），已在 `check_file_name` 中被保护禁止外部访问。

目前文件没有自动清理机制，上传的文件会永久占用磁盘空间。

## Goals / Non-Goals

**Goals:**
- 对 `files/` 目录下的用户上传文件实现基于存留时间的自动过期删除
- 过期时间通过环境变量 `FILE_EXPIRATION_SECONDS` 配置，默认为 `SOFT_TIME_LIMIT` 的 10 倍
- 在 `api.py` 的 lifespan 中启动后台清理任务，周期性执行
- 下载时检查文件是否已过期，若过期则返回 404

**Non-Goals:**
- 不处理 `files/fasttask/` 下的系统内部文件
- 不提供文件恢复/回收站机制
- 不提供基于文件大小、数量等维度的清理策略
- 不提供按文件类型/用户维度的差异化过期策略

## Decisions

### 1. 基于文件修改时间（mtime）判断过期

**选择**：使用 `os.path.getmtime()` 获取文件最后修改时间，与当前时间比较判断是否过期。

**理由**：
- 文件系统自带的时间戳，无需额外维护数据库或索引
- 无需修改上传/下载流程，零侵入
- 简单直接，没有一致性维护成本

**备选方案**：在 Redis 中维护文件元数据（上传时间等）。虽然更精确（mtime 可能被 `cp`/`tar` 等操作重置），但增加了复杂度和维护成本。对于 FastTask 的上传场景，文件上传后直接写入磁盘并返回文件名，mtime 就是上传时间，足够准确。

### 2. 清理逻辑放在 lifespan 后台任务中

**选择**：在 `api.py` 的 `lifespan` 中使用 `asyncio.create_task` 启动一个循环清理协程。

**理由**：
- 与 FastAPI 应用生命周期绑定，服务启动后自动开始，服务停止后自动结束
- 不增加额外进程/线程管理负担
- 符合现有代码风格（`lifespan` 已有后台任务模式）

**备选方案**：用 Celery beat 定时任务执行清理。缺点是：`distributed_worker` 节点不运行 Uvicorn，需要额外考虑调度节点。将清理放在 API 进程中更简单可靠，因为 `single_node` 和 `distributed_master` 都运行 API 服务。

### 3. 清理周期固定为每 10 分钟一次

**选择**：后台任务每 600 秒（10 分钟）执行一次文件过期扫描。

**理由**：
- 文件过期粒度不是秒级的，10 分钟延迟可以接受
- 避免频繁扫描带来的 IO 开销
- 硬编码保持简单，无需额外的环境变量

### 4. 过期文件直接删除，无需确认

**选择**：过期文件直接调用 `os.remove()` 删除。

**理由**：
- 用户需求就是"过期后直接删除"
- 不提供回收站/软删除机制，简化实现
- 和 Celery 的 `RESULT_EXPIRES` 行为一致（后端自动清理任务结果）

### 5. 下载接口增加过期检查

**选择**：在 `check_file_name` 中或下载处理逻辑中增加过期检查，过期文件返回 404。

**理由**：
- 防止在两次清理扫描之间用户下载到即将/应该被删除的文件
- 404 语义上表示文件不存在（无论是未上传还是已过期），不暴露多余信息

## Risks / Trade-offs

- **风险**：进程重启导致清理周期重置。如果服务频繁重启，每次重启后 10 分钟内不会触发首次清理，可能导致过期文件在短时间内存活更久。
  → **缓解**：lifespan 启动时立即执行一次清理，之后按周期循环。

- **风险**：如果 `files/` 目录下文件数量巨大，扫描可能耗时。
  → **缓解**：扫描完成后先 sleep 再循环，避免连续大量 IO。目前项目规模小，`os.listdir` + `os.path.getmtime` 性能足够。

- **风险**：`mtime` 可能被文件复制操作重置。
  → **缓解**：在 FastTask 使用场景中，文件只由 `/upload` 写入一次，不会被再次修改或复制。风险可控。
